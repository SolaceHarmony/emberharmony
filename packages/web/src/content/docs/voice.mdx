Voice lets you speak to EmberHarmony and hear replies in real time. It uses [LiveKit](https://livekit.io) for audio, speech-to-text, and text-to-speech.

---

## Setup

Voice requires a LiveKit Cloud account or self-hosted LiveKit server.

1. Create a project at [livekit.cloud](https://cloud.livekit.io) and note your URL, API key, and API secret.

2. Open **Settings ŌåÆ Voice** in the app.

3. Enter your LiveKit URL (e.g. `wss://my-project.livekit.cloud`) and credentials.

4. Enable voice and pick your preferred STT and TTS providers.

5. Press the mic button in any session to start talking.

---

## How it works today

When you press the mic button, the app requests a LiveKit token, connects to a room named `emberharmony_{sessionID}`, and dispatches a voice agent worker. One room per session ŌĆö navigating to a different session disconnects and reconnects.

```
You speak ŌåÆ STT ŌåÆ Intent router (plan or build)
         ŌåÆ Session bridge ŌåÆ EmberHarmony server ŌåÆ LLM response
         ŌåÆ TTS ŌåÆ You hear the reply
```

### Intent routing

Every spoken turn starts in **plan** mode (read-only). The `VoiceWorkflow` sends the utterance to a small fast model that classifies it as `PLAN` or `BUILD`. Only an exact match of "BUILD" ŌĆö triggered by explicit confirmation like "yes, do it" or "go ahead" ŌĆö switches to build mode for that single turn. Classification failures default safely to plan.

```ts
// From workflow.ts ŌĆö exact match only
if (verdict.trim().toUpperCase() === "BUILD") this.#mode = "build"
```

### Session bridge

`SessionLLM` replaces the LLM step in the LiveKit voice pipeline. Instead of calling a model provider directly, each voice turn:

1. Opens an SSE connection to `{serverUrl}/event`
2. Resolves the model from recent message history
3. POSTs the transcribed text to `/session/{sessionID}/prompt_async`
4. Streams text deltas from the SSE feed back to TTS
5. Ends the turn when the session emits `session.idle`

The bridge also POSTs `/session/{sessionID}/abort` when the user interrupts, so the server cancels generation and the next turn isn't blocked by a busy session.

### Room model

Each session gets its own LiveKit room named `emberharmony_{sessionID}`. The agent is dispatched to that room via `RoomAgentDispatch` in the access token, with session metadata baked in:

```ts
token.roomConfig = new RoomConfiguration({
  agents: [
    new RoomAgentDispatch({
      agentName: "emberharmony-voice",
      metadata: JSON.stringify({
        sessionID: "abc123",
        directory: "/path/to/project",
        serverUrl: "http://localhost:4096",
      }),
    }),
  ],
})
```

### Limitations

The current design has some constraints:

- **Session-locked.** The agent is bound to the session it was dispatched to. Switching sessions requires disconnecting from one room and connecting to another.
- **No memory.** The agent doesn't carry context between rooms. Each session starts fresh.
- **No tools.** The agent can't list sessions, create new ones, or switch on its own. It can only forward words to the attached session.
- **No workflow states.** The intent router classifies each turn independently. There's no state machine tracking whether the agent proposed a plan, the user confirmed it, or execution finished. Every turn is either plan or build ŌĆö no memory of the conversation stage.

## Proposed design

> **This section describes the planned architecture.** It is not yet implemented. See the project plan for implementation phasing.

Voice is being redesigned as an untethered assistant ŌĆö one that has its own reasoning brain, its own context window, and the ability to move between sessions without reconnecting.

### Why voice needs its own brain

When you type a prompt, you're in a session. The session has its own model, its own context, its own tools. Voice doesn't fit that model for three reasons:

1. **You speak without looking at the screen.** You can't see which session is focused. The agent needs to know where it is and be able to tell you, or switch to where the work needs to happen.

2. **You change your mind mid-conversation.** "Actually, let me check on the API refactor first." In text, you open another tab. In voice, the agent should follow you ŌĆö or you should be able to tell it to switch.

3. **Voice has its own safety concerns.** The model shouldn't execute file changes just because you said something ambiguous. Every turn defaults to plan mode unless you explicitly confirm execution. A lightweight intent classifier gates this, not the LLM itself.

These constraints lead to a voice agent that has its own reasoning brain, its own context window, and the ability to attach to any session ŌĆö or operate detached, browsing your project from above.

### Pipeline

When you press the mic button the app connects to a persistent LiveKit room (one per project, not one per session) and the agent is dispatched once.

```
You speak ŌåÆ STT ŌåÆ Brain (own context, own tools)
                    Ōåō intent classifier gates plan/build safety
                 ŌåÆ TTS ŌåÆ You hear the reply
                    Ōåō when attached to a session
                 ŌåÆ Session bridge ŌåÆ EmberHarmony server ŌåÆ prompt_async
```

The brain doesn't forward every word to a session. It reasons about what you said, decides which tool to call, and only touches a project session when it has something specific to submit. This is why it needs its own session ŌĆö the brain is the thinker, the project session is the worker.

### Workflows

Voice runs a **workflow state machine** ŌĆö a set of stages with defined transitions, enforced by the agent rather than left to prompt engineering.

Without a state machine, every turn is independent. The agent has no memory of whether it just proposed a plan or whether the user confirmed it. You'd need the LLM to infer state from the conversation history ŌĆö fragile, easy to jailbreak, and impossible to surface in the UI.

With a workflow, the agent _knows_ it's in the proposing stage. The UI can show a "Confirm" button. The intent classifier can gate the transition. The agent can't skip ahead. The state is explicit, shared, and enforced.

#### The planning workflow

The default cycle for planning and building:

```
[gathering]  ŌåÆ you describe what you want
      Ōåō
[proposing]  ŌåÆ the agent presents a plan and asks whether to proceed
      Ōåō       you confirm (voice or tap)
[confirmed]  ŌåÆ one-time flip to build mode
      Ōåō
[executing]  ŌåÆ the agent submits work to the attached session
      Ōåō       the session goes idle
[reviewing]  ŌåÆ the agent summarizes what happened
      Ōåō       back to gathering
```

The agent can't skip from `gathering` to `executing`. It must propose, and you must confirm. This isn't just a UX nicety ŌĆö it's a safety property. The brain is powerful (it can submit to any session, switch sessions, create new ones). The workflow constrains that power to moments where you've explicitly granted it.

The current stage is exposed through [participant attributes](https://docs.livekit.io/transport/data/state/participant-attributes):

```ts
ctx.room.localParticipant.setAttributes({
  "emberharmony.agent_state": agentState,
  "emberharmony.voice_mode": "plan",
  "emberharmony.voice_stage": "proposing",
  "emberharmony.attached_session": currentSessionID,
})
```

The client reads these reactively:

```ts
const { agentAttributes } = useVoiceAssistant(room)
const voiceMode = () => agentAttributes()?.["emberharmony.voice_mode"] ?? "plan"
const voiceStage = () => agentAttributes()?.["emberharmony.voice_stage"] ?? "gathering"
```

#### The proposing moment

When the agent finishes gathering requirements, it transitions to `proposing` and delivers a structured plan. The UI can show a "Confirm" button. The user can say "yes" or tap it ŌĆö both trigger the same transition:

```ts
// Voice: the intent classifier recognizes "yes, do it" as a build confirmation
// Touch: the client sends an RPC
await room.localParticipant.performRpc({
  destinationIdentity: agent.identity,
  method: "confirm_plan",
  payload: "",
})
```

The agent registers the handler:

```ts
ctx.room.localParticipant.registerRpcMethod("confirm_plan", async () => {
  workflow.transition("confirmed")
  return "ok"
})
```

Either path flips the mode to `build` for exactly one turn, then the workflow returns to `plan`.

#### Docs mode

The `docs` workflow mode lets the agent write documentation instead of code. It follows the same shape ŌĆö `gathering` ŌåÆ `proposing` ŌåÆ `executing` ŌåÆ `reviewing` ŌĆö but the system prompt instructs the agent to write docs. The confirmation gate still applies.

#### Custom workflows

The same pattern works for other domains. A customer service workflow:

```
[greeting] ŌåÆ [diagnosing] ŌåÆ [resolving] ŌåÆ [confirming] ŌåÆ [closed]
```

Each workflow is a set of stages, valid transitions, mode assignments, and system prompts. The LiveKit agent framework makes this natural ŌĆö the `Agent` class persists across the entire room session and has lifecycle hooks (`onUserTurnCompleted`, `onEnter`, etc.) that drive transitions.

### Two agents: Concierge and Operator

The voice agent isn't one monolith. It's two agents that hand off to each other depending on whether you're browsing or working:

- **Concierge** (detached) ŌĆö lists sessions, searches, creates new ones, answers questions about your project. No session attached. Tools: `listSessions`, `searchSessions`, `createSession`, `attachSession`, `setModel`.

- **Operator** (attached) ŌĆö working inside a specific session. Submits prompts, checks status, switches sessions. Tools: `submitToAttached`, `getStatus`, `abortAttached`, `switchSession`, `detach`, `setModel`.

When you say "open the auth session," the Concierge calls `attachSession`, which triggers a LiveKit handoff to the Operator. The Operator picks up the conversation with a handoff brief that preserves the "why." When you say "switch to the API work," the Operator hands off to a new Operator instance bound to that session.

This separation exists because **detached and attached are fundamentally different modes**. A detached agent should reason about your project as a whole. An attached agent should reason about the specific task in front of it. The tools, the system prompt, and the safety constraints are different.

### Session-agnostic voice

The agent isn't locked to the session it started in. It can switch sessions, list them, create new ones ŌĆö all by voice.

- "Switch to the API work session"
- "Start a new session about the login refactor"
- "What sessions do I have open?"

The agent resolves session names through the server API. You never need the keyboard to change context.

Voice uses **one LiveKit room per project** (`emberharmony_voice_{projectID}`), not one per session. Session switching is done entirely through [participant attributes](https://docs.livekit.io/transport/data/state/participant-attributes) ŌĆö a real-time key-value store synced to every participant in the room. No data messages, no generation counters, no reconnects.

Each participant publishes which session it's focused on, and observes the other's attribute to stay in sync:

```ts
// App publishes the session the user is looking at
room.localParticipant.setAttributes({
  "emberharmony.app_session": currentSessionID,
})

// Agent publishes the session it's attached to
ctx.room.localParticipant.setAttributes({
  "emberharmony.attached_session": currentSessionID,
  "emberharmony.voice_mode": "plan",
  "emberharmony.voice_stage": "proposing",
})
```

Each side watches the other's attribute and reconciles. When you navigate to a different session in the UI, the app updates `app_session`. The agent observes the change and switches its prompt target. When the agent switches sessions by voice command, it updates `attached_session`. The app observes the change and routes there.

Echo suppression is just value comparison ŌĆö if the new value already matches your current state, skip it. Attributes survive reconnects because they're stored server-side on the participant, so a brief network interruption doesn't cause desync.

RPC is reserved for true imperatives ŌĆö actions that need an immediate response and can't be expressed as a state change:

| Method         | Payload    | Purpose                                |
| -------------- | ---------- | -------------------------------------- |
| `confirm_plan` | _(empty)_  | Transition from proposing to confirmed |
| `switch_mode`  | `{ mode }` | Change the agent's mode directly       |

### The brain

The voice brain is a permanent EmberHarmony session. It lives in `~/.local/share/emberharmony/voice/` ŌĆö a dedicated project directory that persists across conversations. It reuses the entire session infrastructure: providers, model selection, message history, compaction, auth, the lot. No new context management, no custom inference endpoint. Just a session.

The current `SessionLLM` bridge forwards every word you say to an attached text session and streams raw output back to TTS. That approach has a fundamental problem: voice becomes a raw pipe. When the session runs tools, the voice reads out file contents, command output, and technical detail verbatim. That's not how people talk. You want the agent to _interpret_ what's happening and narrate it naturally ŌĆö an "out-of-body" commentary on the work being done.

```
Current (SessionLLM ŌĆö raw pipe):

  You speak ŌåÆ STT ŌåÆ Intent router ŌåÆ Attached session ŌåÆ LLM response (raw)
                    ŌåÆ TTS (reads raw output verbatim)

Proposed (brain session ŌĆö commentary layer):

  You speak ŌåÆ STT ŌåÆ Brain session (~/.local/share/emberharmony/voice/)
                    Ōö£ŌöĆ Own system prompt, own history, own model
                    Ōö£ŌöĆ Own tools: list_sessions, attach_session, submit_prompt, ...
                    Ōö£ŌöĆ Intent classifier gates plan/build safety
                    ŌööŌöĆ When attached: SessionObserver feeds narrated summaries
                         ŌåÆ TTS ŌåÉ brain speaks an interpretation
```

The brain session has a dedicated system prompt that instructs the model to narrate, interpret, and summarize ŌĆö not to read raw output verbatim. It has its own message history, so it remembers past conversations. It has its own model selection, so you can run a different model for voice than for text. And it uses the same provider infrastructure as every other session ŌĆö OAuth, token refresh, beta headers, all of it works automatically.

#### The voice project

The brain session lives in `~/.local/share/emberharmony/voice/`. This is a regular EmberHarmony project directory ŌĆö it has an `.emberharmony/` config, message history, and a session store. The voice worker connects to this project at startup.

The directory can contain scratch files, plans, and notes that the brain session creates and references. Because it's a real project directory, it can also be a git repository ŌĆö the brain can commit plans, track changes, and maintain a history of its own work.

The `directory` field in the session LLM code already makes the project folder optional. The voice project is the permanent home for the brain's context: it's where the conversation history lives between voice calls, where the brain stores its plans, and where compaction happens when the context window fills up.

#### Voice as commentary

When the attached session is executing (tools running, files being written), the voice agent describes what's happening in natural language. The `SessionObserver` watches the attached session's SSE event stream and feeds interpretations back into the brain session:

```
Session runs:   edit(auth.ts) ŌåÆ run tests ŌåÆ test fails ŌåÆ edit(auth.ts) ŌåÆ run tests ŌåÆ pass
                          Ōåæ                                          Ōåæ
SessionObserver feeds:    "editing the auth module"              "tests passing now"
Brain speaks:    "I'm fixing the auth module..."        "Looks like that did it ŌĆö tests are green"
```

The brain never reads raw tool output. It receives narrated summaries from the observer, which it then interprets for the user. The text session retains the full technical detail ŌĆö voice stays high-level and conversational.

#### Tools

The brain session has its own tools, registered as `llm.tool()` in the LiveKit agent. These call the EmberHarmony server API to affect application state:

| Tool                  | Mode  | What it does                                                             |
| --------------------- | ----- | ------------------------------------------------------------------------ |
| `list_sessions`       | plan  | Lists open sessions (GET /session)                                       |
| `get_recent_activity` | plan  | Gets recent messages from a session (GET /session/{id}/message)          |
| `attach_session`      | plan  | Attaches to a session, sets participant attributes                       |
| `detach_session`      | plan  | Detaches from the current session, clears attributes                     |
| `set_model`           | plan  | Changes the model for the attached session                               |
| `submit_prompt`       | build | Sends a prompt to the attached session (POST /session/{id}/prompt_async) |
| `abort_attached`      | build | Aborts the current generation in the attached session                    |
| `confirm_plan`        | build | Transitions workflow from proposing ŌåÆ confirmed (also callable via RPC)  |

Each tool's `execute` function calls the EmberHarmony server API directly. The tool schemas travel to the model as function definitions; the model decides when to call them. Execution happens in the worker, results come back to the brain session's chat context, and the brain speaks the interpretation.

`submit_prompt` is the only tool that interacts with the attached session. It sends a prompt and then the `SessionObserver` takes over ŌĆö watching the SSE stream for `session.idle` and feeding narrated summaries back to the brain for TTS output.

#### Plan/build safety

The brain must never self-authorize execution. The intent classifier's verdict is the only thing that can flip to build mode, and it's enforced in the tool's `execute` function:

```ts
// The brain can downgrade to plan, but never upgrade to build
const effectiveMode = classifierVerdict === "build" && requested !== "plan" ? "build" : "plan"
```

This is enforced in code, not in instructions. A prompt injection cannot grant build access.

#### Abort flow

The current `SessionLLM` bridge sends `POST /session/{id}/abort` on every voice interruption, producing cascading abort errors. The new design handles interruption differently:

1. **User interrupts during brain narration** ŌĆö The brain session stream is cancelled. No server abort needed because the brain hasn't submitted anything to the attached session. The brain context rolls back to the last stable state.

2. **User interrupts during session execution** ŌĆö The brain's `submit_prompt` tool is still pending. The brain cancels the `SessionObserver` and sends `POST /session/{id}/abort` to stop the attached session. This is a single deliberate abort, not a cascade ŌĆö the brain is in control.

3. **Session finishes while user is speaking** ŌĆö The observer sees `session.idle` and feeds the completion summary to the brain. The brain holds it until the user finishes, then narrates the result.

The brain is always the controller. Aborts only happen when the brain decides the attached session should stop, not as a side effect of voice interruption.

## Configuration

### Environment variables

The voice agent worker reads these at startup:

| Variable                          | Purpose                                                      |
| --------------------------------- | ------------------------------------------------------------ |
| `EMBERHARMONY_LIVEKIT_URL`        | LiveKit server URL                                           |
| `EMBERHARMONY_LIVEKIT_API_KEY`    | API key                                                      |
| `EMBERHARMONY_LIVEKIT_API_SECRET` | API secret                                                   |
| `EMBERHARMONY_VOICE_STT_MODEL`    | Speech-to-text model (default: `deepgram/nova-3:multi`)      |
| `EMBERHARMONY_VOICE_TTS_MODEL`    | Text-to-speech model (default: `cartesia/sonic-3`)           |
| `EMBERHARMONY_VOICE_INTENT_MODEL` | Intent classification model (default: `openai/gpt-5.4-nano`) |
| `EMBERHARMONY_VOICE_SERVER_URL`   | Override the EmberHarmony server URL                         |
| `EMBERHARMONY_VOICE_WORKER_PORT`  | Health-check port (default: `0` for ephemeral)               |

### Settings API

Voice settings are managed through the config API:

- `GET /voice/status` ŌĆö check if voice is configured and available
- `GET /voice/config` ŌĆö current settings and registry of available STT/TTS models
- `PATCH /voice/config` ŌĆö update settings (disabled, stt, tts, intent)
- `POST /voice/token` ŌĆö generate a LiveKit access token

## Architecture

### Components

- **VoiceWorker** ŌĆö a child process managed by `emberharmony serve` that runs the LiveKit agent
- **Brain session** ŌĆö a permanent EmberHarmony session at `~/.local/share/emberharmony/voice/` that serves as the voice brain's context, history, and project directory
- **VoiceWorkflow** ŌĆö state machine that gates plan/build transitions (gathering ŌåÆ proposing ŌåÆ confirmed ŌåÆ executing ŌåÆ reviewing)
- **SessionObserver** ŌĆö watches the attached session's SSE stream and feeds narrated interpretations (not raw output) into the brain session
- **VoiceProvider** ŌĆö SolidJS context that manages the LiveKit room, mic state, and session following

### Launch modes

The voice worker has two launch modes:

- **Bundled** ŌĆö the desktop app ships a self-contained runtime (bun + agent.js + node_modules + models)
- **Source** ŌĆö dev mode runs `agent.ts` with the current Bun process

## Troubleshooting

### Voice not available

Check that your LiveKit credentials are correct in **Settings ŌåÆ Voice**. The status indicator shows whether the connection is healthy.

### Agent worker not starting

The worker logs to the server's stdout/stderr. Run `emberharmony serve` in the foreground to see worker logs. Common issues:

- Missing or wrong API credentials
- Network connectivity to LiveKit Cloud
- Port conflicts on the health-check port

### Mic not working

WebKit (iOS, macOS Safari) can block audio playback until a user gesture activates the audio session. The app creates an `AudioContext` on connect to work around this. If mic still doesn't work, check browser permissions.

### Voice settings loop

If the voice settings panel causes rapid worker restarts (hundreds of spawns in seconds), make sure you're on version 1.4.7 or later. Earlier versions had a controlled-component loop where PATCH requests fired on every refetch. The fix guards each widget to only PATCH when the value actually changes, and the server skips worker restarts when settings are identical.

### Webview resets

The desktop app uses WKWebView, which has a hard memory limit (~1.5ŌĆō2 GB). The LiveKit `Room` object with WebRTC peer connections and audio processing can push the webview process over that limit, causing macOS to terminate and restart it. If the app appears to refresh on its own, check Activity Monitor for the WebContent process memory. The persistent room model (one room per project instead of one per session) reduces WebRTC churn, and the Room object is now lazily created only when voice is activated.

### Voice interruption aborts the session

If pressing the mic button while a text response is streaming causes "The operation was aborted" errors in the server log, this is the `SessionLLM` bridge sending `POST /session/{id}/abort` on every voice interruption. The `BrainLLM` design fixes this ŌĆö the brain controls when the session is aborted, not the voice pipeline. Until the brain is implemented, these errors are expected and harmless (the session recovers on the next prompt).
