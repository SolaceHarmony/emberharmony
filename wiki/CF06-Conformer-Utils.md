# CF06 · Conformer utils (streaming/SD)
**Code:** `CF06` · **Source:** `model/conformer/utils.py` · **Rust:** `model/conformer/utils.rs` · **On the LFM2-Audio inference path:** yes

## Role
A 112-line grab-bag of NeMo-derived helpers used by the FastConformer encoder: an autocast-dtype guard (`avoid_float16_autocast_context`), a streaming-config dataclass (`CacheAwareStreamingConfig`), and a stochastic-depth drop-probability schedule (`compute_stochastic_depth_drop_probs`). None of the three carry tensors; they configure or guard the conformer rather than transform activations. It exists because the conformer encoder/MHA are lifted nearly verbatim from NeMo, which factors these cast/streaming/regularization utilities out of the layer code. On the LFM2-Audio *offline* inference path only the autocast guard is actually invoked (by `mha.py`), and on that path it degenerates to a no-op (see below).

## How it works

**`avoid_float16_autocast_context()` (`utils.py:25-40`).** Returns a *context manager*, not a value. The decision tree: if PyTorch autocast is currently enabled AND the active GPU autocast dtype is `torch.float16` (`utils.py:31`), then it returns a *replacement* autocast that swaps f16 for something safer — `torch.float32` under JIT scripting/tracing (`utils.py:32-33`), `torch.bfloat16` if the device reports bf16 support (`utils.py:35-36`), else `torch.float32` (`utils.py:38`). In every other case (autocast off, or autocast on but already bf16/f32) it returns `contextlib.nullcontext()` (`utils.py:40`) — i.e. no dtype override. The intent is purely defensive: f16 matmuls inside relative-position attention overflow/underflow, so NeMo forces the enclosed block out of f16. It is the only symbol here that touches the forward path: `mha.py:270` and `mha.py:395` wrap `forward_qkv` + the attention score computation in `with avoid_float16_autocast_context():`. Crucially, immediately before that the MHA also does an *explicit* upcast gated on autocast state — `if torch.is_autocast_enabled(): query,key,value = …to(torch.float32)` (`mha.py:266-267`). So on the LFM2-Audio offline path, where the model is run in plain bf16/f32 with **no** `torch.autocast` context active, both `torch.is_autocast_enabled()` and the f16 branch are false: the explicit upcast is skipped and `avoid_float16_autocast_context()` returns `nullcontext()`. The conformer attention therefore runs in the ambient model dtype, and the realized f32 upcast that matters for numerics lives inside `mha.py`'s own score path, not here.

**`CacheAwareStreamingConfig` (`utils.py:42-64`).** A pure `@dataclass` of int knobs for cache-aware *streaming* inference: `chunk_size`/`shift_size` (step granularity, can each be a 2-element `[first_step, others]` list), `cache_drop_size`, `last_channel_cache_size` (cache for MHA-style "last channel" layers), `valid_out_len` (count of trailing outputs that match offline mode), `pre_encode_cache_size` + `drop_extra_pre_encoded` (pre-encoder/subsampling cache bookkeeping), and `last_channel_num`/`last_time_num` (how many MHA vs conv layers need caching). It is instantiated by `encoder.py:891` (`setup_streaming_params`), which is part of the streaming/export machinery. On the offline LFM2-Audio path `_create_masks` returns `(None, None)` and streaming is never engaged, so this struct is constructed but its fields don't gate any tensor op.

**`compute_stochastic_depth_drop_probs(num_layers, p, mode, start_layer)` (`utils.py:66-112`).** Builds a per-layer list of drop probabilities for stochastic-depth regularization (a *training* technique — randomly skipping whole ConformerLayers). Validates `0 ≤ p < 1` and `1 ≤ start_layer ≤ num_layers` (`utils.py:93-96`). Layers `[0, start_layer)` get `0.0` (never dropped) (`utils.py:99`). For the remaining `L = num_layers − start_layer` layers (`utils.py:102`): `"linear"` ramps the probability `l/L · p` for `l = 1..L` so the last layer hits the full `p` (`utils.py:105`); `"uniform"` assigns `p` to all (`utils.py:107`); any other mode raises (`utils.py:109-111`). It is called once at encoder construction (`encoder.py:429-431`) with `len(self.layers)` (the 17 ConformerLayers) and stores `self.layer_drop_probs`. For LFM2-Audio inference `stochastic_depth_drop_prob` defaults to `0.0`, so `L`'s branch produces an all-zeros list and no layer is ever dropped.

## Dtypes & shapes
This component is a config/guard layer; it carries **no activation tensors**. The only dtype it reasons about is the *autocast dtype enum*.

| Symbol | Input(s) | Output |
|---|---|---|
| `avoid_float16_autocast_context()` | global autocast state (enabled? + GPU dtype enum) | a context manager: `autocast(bf16/f32)` if active dtype was f16, else `nullcontext()` (no dtype override) |
| `CacheAwareStreamingConfig()` | int / `[int,int]` field values (default 0) | dataclass instance (no tensors) |
| `compute_stochastic_depth_drop_probs(num_layers:int, p:float, mode:str, start_layer:int)` | scalars | `list[float]` of length `num_layers`, values in `[0, p]` |

Promotion note: the *only* dtype effect on the path is indirect — the guard's f16→{bf16,f32} swap would change attention compute dtype, but on the offline LFM2-Audio path autocast is off, so no promotion occurs here; the f32 attention-score upcast lives in `mha.py`. Conformer activations elsewhere remain model dtype (bf16 on Metal / cuda, f32 on Rust CPU).

## Wiring
**Upstream (who configures/invokes this):**
- [conformer_mha](CF02-RelPos-MHA) calls `avoid_float16_autocast_context()` at `mha.py:270,395` to wrap the QKV/attention block. Edge: no tensor — a context-manager handshake guarding the (batch, time, d_model=512, d_k=64) attention compute.
- [conformer_encoder](CF01-Conformer-Encoder) imports and calls `compute_stochastic_depth_drop_probs(len(layers)=17, …)` at `encoder.py:429` and constructs `CacheAwareStreamingConfig()` at `encoder.py:891`. Edge: scalars/config — no tensor.

**Downstream (who consumes this output):**
- [conformer_encoder](CF01-Conformer-Encoder) consumes the returned `list[float]` (length 17, all 0.0 at inference) as `self.layer_drop_probs`, and the `CacheAwareStreamingConfig` instance as `self.streaming_cfg`. Both are config, not tensors.
- [conformer_mha](CF02-RelPos-MHA) consumes the returned context manager, under which the rel-pos attention scores `(batch, heads=8, time1, time2)` are computed.

## Python ↔ Rust
Symbol-level mapping (`utils.py` → `utils.rs`):
- `avoid_float16_autocast_context()` → `avoid_float16_autocast_context(autocast_dtype: Option<DType>, bf16_supported: bool) -> Option<DType>` (`utils.rs:74-81`). **Deliberate divergence:** candle has no implicit autocast, so the port keeps the *decision* but drops the *context-manager* form — it returns the `DType` the enclosed block should run in (`Some(BF16)`/`Some(F32)` for the f16 case, `None` = `nullcontext`/no override). The `torch.jit.is_scripting()/is_tracing()` branch has no candle analog and is folded to "not taken." Documented in `PORT_STATUS.md` ("torch autocast … candle has no autocast; `avoid_float16_autocast_context` is a `// PORT:` no-op") and `PYTHON_VS_RUST.md:181` (accelerator.autocast → candle equivalents). On the offline path the conformer attention already upcasts to f32 explicitly inside `mha.rs`, so this function is effectively dead — it exists for inventory parity.
- `CacheAwareStreamingConfig` (dataclass) → `struct CacheAwareStreamingConfig` (`utils.rs:40-60`), with the Python "int OR 2-element list" fields modeled as `enum IntOrPair { Int(i64), Pair(i64,i64) }` (`utils.rs:14-36`) plus a `.second()` accessor mirroring Python's `cfg.x[1]` access. **Deliberate:** Rust makes the int-vs-pair polymorphism explicit in the type system; the subsampling pre-encoder always reports `[1, subsampling_factor]`, so the `Pair` branch is the live one for this model (noted in `utils.rs:9-13`).
- `compute_stochastic_depth_drop_probs` → `compute_stochastic_depth_drop_probs(num_layers, p, mode, start_layer) -> Vec<f64>` (`utils.rs:86-120`), a faithful 1:1 port; Python `ValueError` becomes `assert!`/`panic!`. Word-count audit logs this file at **0.55×** Python (small helpers, expected) in `PYTHON_VS_RUST.md:249`.

All three are flagged "off the offline forward path … ported 1:1 for inventory completeness" in the Rust module header (`utils.rs:1-5`).

## Precision / gotchas
- **The autocast guard is a no-op on the LFM2-Audio offline path.** Both the f16 branch in `utils.py:31` and the paired explicit upcast in `mha.py:266-267` are gated on `torch.is_autocast_enabled()`, which is false when the model is run in plain bf16/f32 with no `torch.autocast` block. The numerically-load-bearing f32 upcast for attention scores therefore lives in `mha.py`/`mha.rs`, not here. Do not mistake this helper for the source of conformer attention precision.
- **Stochastic depth is inference-inert.** With the default `stochastic_depth_drop_prob = 0.0`, the linear branch yields `l/L · 0 = 0` for every layer, so `layer_drop_probs` is all zeros and the 17-layer drop logic never fires. It only matters during training.
- **`start_layer` is 1-based and inclusive on both ends** (`1 ≤ start_layer ≤ num_layers`, `utils.py:95`); off-by-one care: layers `[0, start_layer)` are the never-dropped prefix, and the linear ramp indexes `l = 1..=L` so the *final* layer (not the first droppable one) receives the full `p`.
- **`chunk_size`/`shift_size`/`pre_encode_cache_size` are int-or-list.** The Rust `IntOrPair` makes this explicit; for this model the subsampling pre-encoder reports `[1, subsampling_factor]`, so reads go through the `Pair`/`.second()` branch. Treating these as plain scalars (the `Int` branch) would be wrong for this configuration — though moot offline, since streaming is never engaged (`_create_masks` returns `(None, None)`).
- No EOAudio/special-token or quantization concerns apply to this file — it carries no codes and no embeddings.
